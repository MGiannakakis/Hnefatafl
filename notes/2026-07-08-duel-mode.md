# Duel mode: two networks, one rivalry

**Date:** 2026-07-08
**Topic:** Phase 1b — adversarial co-training (`mode=duel`).

← [self-play training](2026-07-08-self-play-training.md) · [overview](2026-07-08-codebase-overview.md)

## Why

Phase 1's "self-play" opponent is a frozen copy of the same network forced to play
the side it never trained — effectively a random player, so no arms race happens.
Duel mode trains **two separate networks** — one always attacker, one always
defender — that improve by beating each other.

## How it works

Alternating phases ([training/duel.py](../training/duel.py)): ATK trains for
`steps_per_phase` timesteps while a frozen copy of DEF plays inside its envs; then
they swap. Within a phase the learner faces a fixed opponent (what PPO expects);
across phases both keep getting stronger. The very first opponent is the untrained
DEF network — approximately random through the action mask, but the same machinery
as every later phase. There is deliberately no random-opponent code path.

Every snapshot encodes boards from **its own side's perspective** (side flag =
its side), so each network only ever sees inputs it was trained on.

## Reading the results

- Each side has its own dashboard (ATK on `duel.dashboard_port`, DEF on +1) and its
  own checkpoints under `checkpoints/<run>/atk/` and `def/`.
- `ep_rew_mean` here means "win rate minus loss rate **against the rival's last
  snapshot**" — hovering near 0 can mean both sides improving in lockstep. For
  absolute progress, evaluate against the fixed random yardstick
  (`mode=eval`, baselines: ATK 16% / DEF 41% win rate vs random).
- To replay the rivalry between any two checkpoints:
  `mode=eval eval.ckpt=<one side> eval.side=<its side> eval.opponent_ckpt=<other side>`.

## Watch out for

- **Cycling**: head-to-head evals oscillating phase over phase (A beats B, B' beats
  A', ...). Fix if it appears: opponent pools (not built yet, on purpose).
- **Mutual stalling**: both sides converging on draws gives zero gradient (draws pay
  0); a small attacker-only draw penalty is the designated future knob.
- **One-sided collapse**: one side winning ~everything for several consecutive
  phases → shorten `steps_per_phase` so the loser catches up more often.
