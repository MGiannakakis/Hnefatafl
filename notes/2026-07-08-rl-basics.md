# RL basics, using this project's vocabulary

**Date:** 2026-07-08

← Back to [overview](2026-07-08-codebase-overview.md) · Next: [the game environment](2026-07-08-game-environment.md)

## Summary

Reinforcement learning (RL) is learning by trial and error. There's no teacher showing correct moves. Instead, the program plays, occasionally receives a numeric score (**reward**), and slowly adjusts itself so actions that historically led to higher scores become more likely. This note maps each textbook RL word onto the actual thing in this repo.

## The cast of characters

- **Agent** — the learner. Here: a neural network that receives a board and outputs a move. It plays *one side only* (attackers or defenders).
- **Environment** — the world the agent acts in. Here: the Hnefatafl game itself, wrapped in `TaflEnv` ([env/tafl_wrapper.py](../env/tafl_wrapper.py)). Crucially, **the opponent is considered part of the environment** — from the agent's point of view, "I move, then the world changes somehow (the opponent moved), then it's my turn again."
- **Observation** — what the agent sees each turn. Here: the board encoded as a 9×9 grid with 6 layers of numbers (where the attackers are, where the defenders are, where the king is, ...). Details in the [environment note](2026-07-08-game-environment.md).
- **Action** — one move. Here: an integer from 0 to 1295, each encoding "move the piece on square X to square Y."
- **Reward** — the score. Here it's sparse: roughly **+1 for winning, -1 for losing, ~0 for everything in between**. The agent gets almost no feedback during the game — only at the end does it learn whether its choices were good.
- **Episode** — one complete game, from starting position to win/loss/draw.
- **Policy** — the agent's strategy: a function from *board* to *probabilities over moves*. This is exactly what the neural network computes. "Training" means "improving the policy."

## How can it learn from a reward that only comes at the end?

Two ideas make this work:

1. **Credit assignment via discounting.** When a game is won, *all* the moves in that game get some credit, but earlier moves get slightly less (each step back multiplies credit by `gamma = 0.99`). Over thousands of games, moves that genuinely cause wins keep getting reinforced, while moves that just happened to be nearby average out.
2. **A learned value estimate (the "critic").** Alongside choosing moves, the network also learns to answer "from this board position, what's my expected final reward?" — a number like +0.6 meaning "I'm probably winning." This lets the algorithm judge each move by whether it made the position *better than expected* (called the **advantage**), which is a much richer training signal than the raw win/loss alone.

## PPO in plain English

The learning algorithm is **PPO (Proximal Policy Optimization)** — specifically `MaskablePPO` from the `sb3-contrib` library. The loop:

1. **Play** a few thousand moves with the current policy, recording (board, move chosen, reward received, value estimate) for each step.
2. **Score** each recorded move: did it work out better or worse than the critic expected?
3. **Nudge** the network: increase the probability of better-than-expected moves, decrease worse-than-expected ones.
4. The "Proximal" part: **nudge only a little**. PPO clips each update so the new policy can't lurch far from the old one in a single step. Big lurches are how RL training collapses; small clipped steps keep it stable.
5. Throw away the recorded data (it describes the *old* policy's behavior) and go back to step 1.

You don't implement any of this by hand — the library `stable-baselines3` (SB3) does. This repo's job is to supply the environment, the network architecture, and the self-play choreography around it.

## The "Maskable" part: only legal moves

Of the 1296 possible action numbers, only a handful are legal in any given position. `MaskablePPO` accepts an **action mask** — a list of 1296 true/false flags — and forces the illegal moves' probabilities to zero before choosing. Without this, the agent would waste most of its early life being punished for trying impossible moves. The environment provides the mask via `action_masks()` in [env/tafl_wrapper.py:140](../env/tafl_wrapper.py#L140).

## Self-play: where does the opponent come from?

A fixed opponent creates a fixed puzzle — the agent would learn to exploit *that* opponent and stop improving. **Self-play** instead pits the agent against a frozen snapshot of itself from the recent past, refreshed periodically. The opposition automatically scales with the agent's own skill, producing a curriculum that goes from "both of us are terrible" to "both of us are decent" without any human designing it. The mechanics are in the [self-play training note](2026-07-08-self-play-training.md).

## Jargon cheat-sheet

| Term you'll see in code/logs | Plain meaning |
|---|---|
| `policy` | The network's strategy: board in → move probabilities out |
| `rollout` | A batch of recorded gameplay used for one learning update |
| `timestep` | One agent move (1M timesteps ≈ a few tens of thousands of games) |
| `ep_rew_mean` | Average final reward of recent games (~win rate signal: +1 win, -1 loss) |
| `ep_len_mean` | Average game length in agent moves |
| `value_loss` | How wrong the critic's "am I winning?" predictions are |
| `entropy` | How random the policy still is (high early = exploring, drops as it commits) |
| `approx_kl`, `clip_fraction` | How hard PPO is pushing the policy per update (stability gauges) |
